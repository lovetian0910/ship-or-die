# 随机事件系统设计

> 模块：随机事件系统
> 状态：设计确认
> 日期：2026-03-17

---

## 一、整体架构

三种事件入口，两种事件体验：

| 入口 | 玩家能动性 | 事件类型 |
|------|-----------|---------|
| 被动危机 | 无法回避，只能应对 | 打（危机中藏机会） |
| 固定探索点 | 主动选择进不进 | 搜70% / 打30% |
| 随机探索点 | 主动选择，有时间窗口 | 搜70% / 打30% |

### 与塔科夫的体验映射

- **探索节点** = 主动选择去某个位置搜刮，不确定里面有什么
- **被动危机** = 行进中被敌人袭击，无法回避只能应对
- **进入探索后可能遇到危机** = 去高价值房间可能碰到重装Boss

### 架构流程

```
研发时间轴推进中
  │
  ├─ 被动危机（阶段密度分配触发，不可回避）
  │    → 危机中藏机会（安全脱身 vs 冒险搏收益）
  │    → 单局 2-4 次
  │
  ├─ 固定探索点（特定研发阶段必定出现）
  │    → 进入后触发随机事件（70%搜 / 30%打）
  │    → 玩家可选择进不进去
  │
  └─ 随机探索点（概率出现，有时间窗口）
       → 进入后触发随机事件（70%搜 / 30%打）
       → 过期消失，玩家需即时决策要不要进
```

## 二、搜类事件：发现-估值-拿取

### 核心决策

**这个东西值不值得我花时间拿？**

用时间（生命）押注一个确定的收益——风险不在于"能不能拿到"，而在于"花了这些时间之后，后面还够不够用"。

### 事件子类型

| 子类型 | 收益方向 | 代价 |
|--------|---------|------|
| 人才发现 | 品质上限提升 | 时间 |
| 技术方案 | 时间效率提升 | 时间 |
| 差异化方向 | 品质加成 + 竞争优势 | 时间 |
| 资源机会 | 获得事件应对资源 | 时间 |

### 设计要点

- 收益对应入场资源三类功能（品质上限 / 时间效率 / 事件应对），让搜到的东西能补强短板
- 代价统一用时间计价，估值决策始终回到"时间够不够"的核心焦虑
- 不同事件的时间代价有差异，大收益 = 大时间消耗，但比例不线性——有的性价比高有的低，需要玩家判断
- 收益和代价全部明示，无随机判定

## 三、打类事件：危机中藏机会

### 核心决策

**安全脱身还是冒险搏收益？**

被动危机不是纯负面惩罚，而是"风险中藏着机会"——高风险选项可以把危机变成收益来源。

### 事件子类型

| 子类型 | 事件性质 | 典型场景 |
|--------|---------|---------|
| 团队危机 | 内部人员问题 | 人被挖走、团队内讧、核心成员摸鱼 |
| 技术事故 | 技术层面炸了 | 服务器崩了、代码丢了、引擎出bug |
| 外部冲击 | 外面出事了 | 竞品抢跑、平台政策变动、舆论风波 |
| 资源意外 | 资源出问题 | 钱烧完了、外包跑路、素材侵权 |

### 统一选项结构

每个打类事件提供 2-3 个选项：

| 选项层级 | 对应塔科夫 | 代价 | 收益 |
|---------|-----------|------|------|
| 保守选项（永远可选） | 逃跑 | 承受确定惩罚，省时间 | 无 |
| 稳妥选项（可能需要资源） | 正面交火 | 消耗事件应对资源 | 消除惩罚 |
| 冒险选项（需要资源+时间） | 击杀后搜身 | 消耗资源 + 大量时间 | 消除惩罚 + 额外收益 |

### 设计要点

- **保守选项永远存在**：不会出现"没资源就死局"，最差也能割肉脱身
- **冒险选项不是必需的**：给"贪心 + 有资源 + 觉得时间够"的玩家准备的陷阱/机会
- **入场资源决定选项空间**：有事件应对资源 = 多一条路，没有 = 只能保守脱身
- **选项代价和收益全部明示**：不确定性来自全局，不来自单个事件

## 四、触发机制：阶段密度分配

### 设计原则

- 不用概率累积（会导致后期事件扎堆，干扰撤离决策）
- 不用时间轴固定触发（会被摸透规律，削弱突然感）
- 按研发阶段分配事件预算，阶段内部概率触发保持不确定性

### 阶段分配

| 研发阶段 | 时间占比 | 被动危机 | 探索节点 | 设计意图 |
|---------|---------|---------|---------|---------|
| 前期（立项启动） | ~30% | 0-1 次 | 1 个固定探索点 | 相对安全，让玩家搜集资源、建立优势 |
| 中期（核心开发） | ~40% | 1-2 次 | 1 个固定 + 随机探索点 | 压力高峰，搜和打都最密集 |
| 后期（收尾抉择） | ~30% | 0-1 次 | 偶发随机探索点 | 危机减少，留"最后的诱惑" |

### 机制细节

- **阶段内部概率触发**：保持不确定性，玩家不知道具体什么时候来
- **每阶段独立预算**：不跨阶段累积，不会出现前面攒概率后面爆发
- **触发后冷却**：刚处理完一个事件，短时间内不会立刻来第二个
- **后期留白**：危机基本结束，安静本身就是压力（时间在流逝，市场在被蚕食），偶发的随机探索点是"最后的诱惑"

### 单局事件总量

| 来源 | 数量 | 类型 |
|------|------|------|
| 被动危机 | 2-4 次 | 打（危机中藏机会） |
| 固定探索点 | 2 次 | 搜70% / 打30% |
| 随机探索点 | 0-2 次（看玩家选择） | 搜70% / 打30% |
| **总计** | **4-8 次** | 搜打混合 |

对应 10 分钟单局、研发阶段约 5-6 分钟，平均每 45-90 秒一次事件决策。

## 五、核心设计原则

### 单个事件内无随机判定

- 选项的代价和收益全部明示
- 不存在"是否成功"的掷骰判定
- 不确定性来自全局状态：

| 确定的（单事件内） | 不确定的（全局层面） |
|-------------------|-------------------|
| 这个选项花多少时间 | 后面还会不会触发被动危机 |
| 这个选项给多少品质 | 市场热度接下来会不会变 |
| 这个选项需要什么资源 | AI 竞品什么时候上线 |

### 事件包装原则

- **影射现实，虚构名称**：碰瓷真实事件/人物/公司/产品，但换名字
- **一看就懂，会心一笑**：不懂行业的人也能理解决策，懂的人额外获得共鸣
- **有画面感**：事件描述角色化、故事化、带荒诞幽默
- 遵守世界观约束：禁止使用现实中的真实地名、产品名、品牌名、公司名

### 包装示例

> "一个穿黑色高领衫的人推门进来，说他要改变世界，但先得借用你的车库。"
>
> "你的游戏被一个叫'蒸汽'的平台下架了，理由是'含有不当的像素数量'。"
>
> "团队有人提议：不如给游戏加个抽卡系统？玩家群炸了，但收入预测直接翻了三倍。"

## 六、备选方案记录

以下方案暂不采用，留作后续调整参考：

| 方案 | 内容 | 暂不采用原因 |
|------|------|-------------|
| 搜类备选：发现-取舍-放弃 | 发现多个选项只能选一个，核心是机会成本 | 情绪偏向遗憾纠结，不够恐惧 |
| 搜类备选：搜到-盲盒-揭晓 | 拿回去才揭晓价值，核心是赌博感 | 削弱玩家能动性，决策无意义 |
| 触发备选：时间轴固定触发 | 按研发进度30%/60%/90%固定触发 | 可被摸透规律，削弱突然感 |
| 触发备选：概率累积触发 | 越久不出事概率越高，触发后重置 | 后期事件扎堆，干扰撤离决策 |
